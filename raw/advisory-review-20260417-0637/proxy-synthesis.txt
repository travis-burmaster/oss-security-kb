## 3. Exploit Precondition Analysis (Conservative Interpretation)

The following preconditions should be evaluated when assessing actual exploitability in a consuming application:

1. **Attacker-controlled input to serialization.** All five advisories assume that an attacker can influence the JavaScript values passed to `serialize()`. If the data being serialized is entirely server-generated and trusted, direct exploitability is substantially reduced (though defense-in-depth still argues for patching).

2. **Output is evaluated or embedded in an executable context.** The XSS and RCE advisories (#1–#4) require that the serialized string is subsequently:
   - embedded in an HTML `<script>` block (typical in SSR / server-rendered pages), **or**
   - passed to `eval()`, `new Function()`, or an equivalent dynamic-evaluation mechanism.
   If the output is only consumed as a static data string (e.g., written to a JSON file and never eval'd), the injection vectors do not apply.
